# PRD 1.0

## 문서 정보
- **제품명**: 리커리어
- **문서 버전**: 1.0
- **작성일**: 2026-06-30
- **기준 자료**: 제품 기획 워크시트, 온보딩 결정, docs/design-system.md, docs/data-guide.md, docs/firebase-guide.md

## 1. 제품 한 줄 설명
리커리어는 고액 정규직 채용이 부담스러운 성장기 스타트업이 검증된 시니어 전문가를 필요한 시간만큼 프로젝트 단위로 연결하고 계약할 수 있게 돕는 채용/협업 플랫폼입니다.

## 2. 핵심 고객
- **1차 고객**: 시드 투자를 받고 시리즈 A 단계로 스케일업하려는 7년 차 제조 기반 IT 스타트업 창업자
- **대표 페르소나**: 스마트 가전 스타트업을 운영하며 글로벌 진출을 앞둔 30대 창업자 박대표
- **집중하는 쪽**: 기업/수요 측

## 3. 문제 정의
글로벌 진출을 앞둔 스타트업 대표는 현지 네트워크와 경험을 가진 임원급 해외영업 인재가 필요하지만, 정규직 채용에 필요한 고액 연봉과 지분 부담 때문에 적절한 인재를 확보하지 못하고 있습니다. 그 결과 채용 공고를 반복해 올리고 내리며 시간을 잃고, 경쟁사에 해외 시장 기회를 빼앗길 위험이 커집니다.

## 4. 가치 가설
성장통을 겪는 스타트업 대표에게 검증된 시니어 전문가를 주 10~20시간 단위로 매칭하고 계약을 중개하면, 기업은 고액 정규직 채용 부담 없이 필요한 역량을 빌려 쓸 수 있고 프로젝트 성사 시 플랫폼 수수료를 지불할 것입니다.

- **지불 주체**: 기업
- **수익 모델**: 성사 수수료, 추후 구독
- **가격 가설**: 프로젝트 성사 시 1회성 15% 플랫폼 수수료

## 5. MVP 목표
MVP는 기업이 해결해야 할 문제를 등록하고, 추천된 시니어 전문가의 역량을 확인한 뒤, 주/월 단위 조건으로 계약까지 진행하는 첫 사용 흐름을 검증합니다.

핵심 검증 질문은 다음과 같습니다.
- 기업은 경력 조건보다 해결할 문제 중심으로 공고를 등록할 수 있는가?
- 기업은 1분 음성 프로필과 경력 요약만으로 전문가에게 미팅을 제안할 만큼 신뢰를 느끼는가?
- 기업은 정규직이 아닌 주/월 단위 계약 조건을 이해하고 결제 또는 계약 의사를 표현하는가?

## 6. 핵심 기능

### 6.1 문제 중심 공고 작성
- 기업은 회사 정보와 해결해야 할 문제를 입력합니다.
- 경력 요구사항보다 문제 상황, 예산 범위, 필요 기간, 주당 예상 시간을 중심으로 작성합니다.
- 등록 후에는 접수 완료 안내를 받습니다.

### 6.2 시니어 전문가 탐색과 1분 음성 프로필
- 기업은 분야, 경력, 조건 필터로 후보 전문가를 탐색합니다.
- 전문가 카드에는 사진, 이름, 직함, 주요 경력, 희망 근무 조건이 보입니다.
- 상세 화면에서는 전문가가 직접 녹음한 1분 음성 프로필을 재생할 수 있습니다.
- 기업은 관심 전문가에게 탐색 미팅을 제안합니다.

### 6.3 주/월 단위 계약 조건 설정
- 기업은 근무 시간, 계약 기간, 예산 범위를 선택합니다.
- 조건을 확정하면 계약 진행 또는 결제 단계로 이동합니다.
- 계약 완료 후 프로젝트 시작 안내와 다음 단계가 표시됩니다.

## 7. 첫 사용 흐름

### 7.1 기업 흐름
1. 회원가입
2. 문제 등록
3. 전문가 리스트 탐색
4. 전문가 프로필 확인과 1분 음성 청취
5. 미팅 제안
6. 계약 조건 설정
7. 계약 완료 및 프로젝트 시작

### 7.2 시니어 전문가 흐름
1. 디지털 역량 인증 및 전문 분야 등록
2. 1분 음성 프로필 등록
3. 문제 중심 공고 탐색 또는 추천 확인
4. 미팅 및 조건 협의
5. 계약 체결
6. 프로젝트 수행과 목표 달성 조율

## 8. 주요 화면

### 8.1 문제 등록 화면
- **진입**: 인테이크 인터뷰 완료 후 문제 등록 링크로 진입
- **구성요소**: 문제 상황 입력, 예산 범위, 필요 기간, 주당 예상 시간
- **주요 버튼**: 문제 등록하기
- **다음 화면**: 접수 완료 안내

### 8.2 전문가 리스트 화면
- **진입**: 문제 등록 후 또는 추천 후보 확인
- **구성요소**: 검색, 분야/경력/예산/시간 필터, 전문가 카드 목록
- **전문가 카드**: 사진, 이름, 직함, 핵심 경력, 희망 조건, 평점 또는 검증 표시
- **다음 행동**: 전문가 상세 보기

### 8.3 전문가 프로필 화면
- **진입**: 전문가 카드 클릭
- **구성요소**: 사진, 이름, 직함, 1분 음성 플레이어, 해결 경험 요약, 희망 근무 조건
- **주요 버튼**: 탐색 미팅 신청
- **다음 화면**: 일정 또는 미팅 제안 화면

### 8.4 미팅 제안 화면
- **구성요소**: 미팅 제안 메시지, 제안 일정, 미팅 방식
- **주요 버튼**: 미팅 제안 보내기
- **다음 화면**: 일정 조율 또는 조건 협의

### 8.5 계약 조건 설정 화면
- **구성요소**: 근무 시간, 계약 기간, 예산 범위, 계약 조건 요약
- **주요 버튼**: 계약 조건 확정하기
- **다음 화면**: 계약서 확인 또는 착수금 결제

### 8.6 계약 완료 화면
- **구성요소**: 계약 완료 안내, 다음 단계 체크리스트, 프로젝트 시작 버튼
- **주요 버튼**: 프로젝트 시작하기, 계약서 및 파일 확인하기

## 9. 역할과 권한
현재 레포의 역할 기반 구조는 유지합니다.
- **recruiter**: 기업/채용담당자 역할로 우선 사용
- **candidate**: 지원자 또는 시니어 전문가 역할로 우선 사용
- **interviewer**: 기존 구조 유지, MVP에서 필요해지는 시점에 세부 역할 재검토

역할 값을 새로 만들거나 이름을 바꾸는 작업은 docs/data-guide.md 기준에 따라 먼저 팀 논의가 필요합니다.

## 10. 디자인 방향
- **레퍼런스**: 원티드긱스, 기업/시니어 전문가 유저 플로우 이미지
- **분위기**: 부드럽고 친근
- **메인 색상**: 파란색 계열 유지
- **화면 원칙**: 숫자 흐름이 바로 보이고, 사용자가 무엇을 하는 곳인지 직관적으로 이해되는 구조
- **선호 요소**: 지원자 또는 전문가 사진이 보이는 화면
- **비선호 요소**: 글자가 작고 빽빽한 화면
- **운영 강도**: 최소 기준을 맞추고, 약한 불일치는 후속 디자인 정리로 보완

## 11. 데이터 기준
- 데이터 이름은 docs/data-guide.md의 공통 이름 사전을 먼저 확인합니다.
- 새 컬렉션, 필드, 상태값, 역할 값이 필요하면 구현 전에 docs/data-guide.md에 먼저 추가합니다.
- 필드명은 영어 camelCase를 사용합니다.
- 상태값은 자유 텍스트가 아니라 정해진 값만 사용합니다.
- 화면 컴포넌트와 데이터 접근 코드는 분리합니다.

MVP 구현 중 새로 논의될 가능성이 큰 데이터는 다음과 같습니다.
- 문제 중심 공고
- 시니어 전문가 프로필
- 음성 프로필
- 미팅 제안
- 계약 조건
- 결제 또는 에스크로 상태

## 12. Firebase 기준
- Firebase 설정값은 .env에만 저장합니다.
- SDK 값은 문서에 기록하지 않습니다.
- 현재는 Hosting 중심으로 사용합니다.
- Firestore, Storage, Functions는 필요해지는 시점에 docs/firebase-guide.md 기준으로 추가합니다.
- 배포 전에는 빌드를 확인합니다.

## 13. MVP 범위

### 포함
- 기업 회원가입 또는 진입 화면
- 문제 중심 공고 등록
- 전문가 리스트와 필터
- 전문가 사진과 프로필 상세
- 1분 음성 프로필 재생 UI
- 미팅 제안
- 계약 조건 선택
- 계약 완료 안내

### 제외
- AI 자동 매칭 알고리즘
- 실제 결제 연동
- 실제 에스크로 정산 자동화
- 복잡한 관리자 페이지
- 실시간 채팅 전체 구현
- 정규직 채용 중심 기능

## 14. 성공 기준
- 사용자가 첫 화면에서 서비스가 무엇을 하는 곳인지 이해할 수 있습니다.
- 기업 사용자가 문제 등록부터 계약 완료까지의 흐름을 끊기지 않고 따라갈 수 있습니다.
- 전문가 사진, 경력 요약, 1분 음성 프로필이 함께 보여 신뢰 형성에 도움을 줍니다.
- 모바일과 데스크톱에서 글자와 버튼이 잘리지 않습니다.
- 역할 기반 SPA 구조를 유지합니다.
- 새 데이터 이름은 공통 이름 사전 확인 후 사용합니다.

## 15. 추후 논의할 항목
- 시니어 전문가를 기존 candidate 역할로 볼지, 별도 명칭이 필요한지
- 문제 중심 공고를 기존 jobPostings로 확장할지, 별도 컬렉션이 필요한지
- 음성 프로필 파일 저장에 Firebase Storage를 사용할지
- 미팅 일정 조율을 수동 입력으로 시작할지, 자동화할지
- 결제와 에스크로를 실제 연동할 시점
- 플랫폼 운영자가 개입하는 범위
