# 프로젝트 협업 규칙

여러 수강생이 같은 레포에서 AI를 통해 협업하기 위한 기본 규칙입니다.

## 목적

- 수강생이 직접 코딩하는 방식이 아니라, AI에게 작업을 맡기는 방식을 정합니다.
- 여러 명이 동시에 기능을 요청해도 공통 구조가 깨지지 않게 합니다.
- PR, 검토, 승인, 병합 기준을 정합니다.

## 기본값

### 역할 분담

- 수강생은 기획, 요구사항, 기준 결정, 결과 검토를 담당합니다.
- AI는 코드 수정, 문서 수정, 테스트 확인, 커밋, PR 생성을 담당합니다.
- AI는 PR 리뷰에서 에러, 공통 구조 훼손, 문서 기준 위반 여부를 확인합니다.
- 팀에서 확정한 기준은 AI가 담당 문서에 반영하고, 사람이 결과를 검토합니다.
- 팀장은 PR 병합을 최종 결정합니다.

### 작업 단위

작업은 아래 단위 중 하나로 구분합니다.

- 지원자 화면
- 채용담당자 화면
- 면접관 화면
- 공통 화면
- 공통 컴포넌트
- 데이터 구조
- Firebase 연결과 배포
- 문서 수정

### 구현 독립성

- 각 작업 단위(유저 플로우)는 가능한 한 **완전히 독립적으로 구현**합니다.
- 다른 사람의 작업 결과에 의존하지 않도록, 비슷한 코드가 **중복 구현되는 것을 감수**합니다.
- 목적은 병합 안전성입니다. 서로 무엇을 작성 중인지 정확히 몰라도, 병합했을 때 충돌이나 깨짐이 생기지 않게 합니다.
- 한 플로우의 구현이 다른 플로우의 파일을 함께 수정해야 하는 구조는 피합니다.
- 예외(중복하지 않고 공유하는 영역): `docs/data-guide.md`의 공통 이름 사전입니다. 데이터 이름은 오히려 공유해야 병합 시 데이터가 맞습니다. 즉 **구현 코드는 독립, 데이터 이름은 공유**가 기준입니다.

### 작업 전 문서 확인

- 모든 작업에서 모든 문서를 매번 확인하지 않습니다.
- 먼저 작업 단위를 구분하고, 해당하는 담당 문서만 확인합니다.
- 데이터 이름, 컬렉션, 필드, 상태값, 역할 값이 필요하면 `docs/data-guide.md`를 먼저 확인합니다.
- 새 데이터 이름이 필요하면 코드 작성 전에 `docs/data-guide.md`의 공통 이름 사전에 먼저 추가합니다.
- 화면 톤, 레이아웃, 공통 컴포넌트가 필요하면 `docs/design-system.md`를 확인합니다.
- 공통 컴포넌트는 문서 기준에 맞는 기존 요소가 있으면 재사용하고, 없으면 새로 만듭니다.
- Firebase 연결, 환경변수, DB 사용, 배포가 필요하면 `docs/firebase-guide.md`를 확인합니다.

### 브랜치

- `main` 브랜치는 배포 가능한 상태로 유지합니다.
- AI 작업은 개인별 작업 브랜치에서 진행합니다.
- 브랜치 이름은 `student/이름/작업명` 형식을 기본으로 합니다.
- 이름은 영문 소문자 또는 Git에서 안전한 표기를 사용합니다.
- 작업명은 화면, 기능, 수정 목적이 드러나게 짧게 작성합니다.
- 작업이 끝나고 PR이 병합된 브랜치는 로컬과 원격에서 삭제합니다.

### PR

- 모든 기능 브랜치는 `main` 병합 전에 PR을 만듭니다.
- PR 생성은 AI가 수행합니다.
- PR 제목, 설명, 변경 요약, 확인한 명령, 남은 이슈는 AI가 작성합니다.
- PR 설명에는 데이터 사전 변경 여부를 적습니다.
- PR은 긴 회의용 리뷰 문서가 아니라 `main` 진입 전 체크포인트로 사용합니다.
- AI는 PR 내용을 리뷰하고 병합 가능 여부를 판단합니다.
- 팀장은 AI 리뷰 결과를 참고해 최종 승인합니다.
- 디자인이 약간 어긋나는 정도는 후속 정리 작업으로 남길 수 있습니다.
- 기본 레이아웃처럼 공통 구조를 깨는 변경은 후속 작업으로 미루지 않습니다.

### AI 리뷰 통과 기준

- 앱 실행 또는 빌드에서 에러가 없어야 합니다.
- 병합 후 깨질 가능성이 큰 충돌이 없어야 합니다.
- SPA 구조를 유지해야 합니다.
- 역할 기반 화면 분기를 훼손하지 않아야 합니다.
- 역할 값, 공통 컴포넌트, 데이터 구조를 이유 없이 바꾸지 않아야 합니다.
- 디자인은 `docs/design-system.md`의 최소 기준을 크게 벗어나지 않아야 합니다.

### 병합

- `main` 병합은 승인 후 진행합니다.
- 여러 기능 브랜치를 한 번에 병합하지 않습니다.
- 기능 브랜치가 여러 개 있으면 PR 단위로 하나씩 순서대로 병합합니다.
- 하나의 PR이 병합된 뒤 다음 브랜치를 최신 `main` 기준으로 확인합니다.
- 충돌이 생기면 AI가 해결안을 제시하고, 사람이 확인합니다.
- 데이터 구조, 디자인 시스템 변경은 추가 확인 후 병합합니다.
- PR 병합은 팀장만 진행합니다.
- PR 병합 후 해당 작업 브랜치는 삭제하고, 남은 브랜치 목록을 확인합니다.

## 논의할 항목

- 공통 구조 변경 승인 기준
- 작업 단위를 어떻게 나눌지
- 동시에 같은 파일을 수정할 때의 우선순위
- AI가 커밋과 PR까지 자동으로 진행해도 되는 범위
- 디자인 정리를 별도 후속 작업으로 남길 수 있는 기준
- 병합 후 브랜치 삭제 담당자
- 데이터 사전 변경 승인 기준

## 최종 결정

- 작업 단위: 역할 화면, 공통 구조, 데이터/Firebase, 문서 수정 단위로 구분
- 구현 독립성: 작업 단위(유저 플로우)는 독립 구현을 기본으로 하고, 중복 구현을 감수해 병합 충돌을 줄임. 단 데이터 이름 사전은 공유
- 작업 전 문서 확인: 작업 성격에 맞는 담당 문서만 확인
- 브랜치 전략: AI 작업은 개인별 작업 브랜치에서 진행하고 `main`은 배포 가능한 상태로 유지
- 브랜치 이름: `student/이름/작업명` 형식 사용
- PR 사용 여부: 필수
- PR 작성 기준: AI가 제목, 설명, 변경 요약, 데이터 사전 변경 여부, 확인 명령, 남은 이슈를 작성
- PR 운영 방식: `main` 병합 전 가벼운 체크포인트로 사용
- PR 승인 기준: AI가 리뷰하고 팀장이 최종 승인
- AI 리뷰 기준: 실행 또는 빌드 에러, 충돌 위험, 공통 구조 훼손, 문서 기준 위반을 우선 확인
- 병합 담당: 팀장만 `main` 병합 진행
- 병합 순서: PR 단위로 하나씩 직렬 병합
- 병합 후 브랜치 정리: 병합된 작업 브랜치는 로컬과 원격에서 삭제
- 공통 구조 변경 기준: 데이터 구조, 디자인 시스템 변경은 추가 확인 후 병합
- 충돌 해결 기준: AI가 해결안을 제시하고 사람이 확인
- AI 커밋/PR 권한: AI가 커밋과 PR 생성을 수행
- 데이터 사전 변경 기준: 새 데이터 이름은 구현 전 `docs/data-guide.md`에 먼저 추가
- 디자인 기준: 최소 기준을 맞추고, 세부 구현은 AI가 보완
- 공통 컴포넌트 기준: `docs/design-system.md`를 먼저 확인하고 기존 요소가 있으면 재사용
- Firebase 기준: DB 사용까지 허용하되 구현 시점에 `docs/firebase-guide.md` 기준으로 구체화
- 문서 반영 담당: AI가 팀 결정사항을 담당 문서에 반영하고 사람이 검토
- 디자인 후속 정리 기준: 약한 불일치는 후속 정리 가능, 공통 구조를 깨는 변경은 후속으로 미루지 않음
